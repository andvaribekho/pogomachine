A1. [P0] SHARED GEOMETRY IS BEING DISPOSED → silent rendering corruption
--------------------------------------------------------------------------------
Three.js geometries/materials kept at module scope are referenced by many
meshes. Calling .dispose() on one of them frees GPU buffers used by ALL
instances. Subsequent draw calls render with stale/garbage GPU memory.

Affected sites:

  • main.js:716 createFloater()
        new THREE.Mesh(floaterDiscGeometry, floaterMaterial.clone())
    main.js:988 updateFloaters()  → floater.mesh.geometry.dispose()
    main.js:1002 updateFloaters() (also disposes when scrolled past)
    main.js:3584 clearTower()     → f.mesh.geometry.dispose()
    => Disposes the SHARED floaterDiscGeometry. After the first floater is
       consumed, all remaining + future floaters render broken.

  • main.js:2530 spawnCoinPickup()
        new THREE.Mesh(coinPickupGeometry, coinPickupMaterial)
    main.js:2555 updateCoinPickups() → pickup.mesh.geometry.dispose()
    main.js:2565 clearCoinPickups()  → pickup.mesh.geometry.dispose()
    => Same issue. Shared coinPickupGeometry destroyed on first collect.

  • main.js:1143 detachAcidPuddlesFromTile()
        new THREE.Mesh(acidDropletGeometry, dropletMaterial)
    main.js:1111 updateAcidPuddles() → puddle.mesh.geometry.dispose()
    main.js:1160 clearAcidPuddles()  → puddle.mesh.geometry.dispose()
    => If a falling droplet is force-cleared, the shared acidDropletGeometry
       is disposed, breaking every later acid drop that level.

FIX:
  Remove every .geometry.dispose() call on instances that share a module-scope
  geometry. Only dispose the per-instance materials (which are cloned).
  For the falling acid droplet, never dispose the shared geometry; only
  dispose the per-instance material.

  Establish a project rule: "If the geometry was constructed at module scope
  (top of main.js block ~lines 291–325), it is SHARED and must NEVER be
  disposed by per-instance cleanup."

================================================================================
PART B — PERFORMANCE
================================================================================

GOALS
  • Lift sustainable FPS on low/mid Android (Snapdragon 6-series, A11/A12 iOS,
    integrated Intel laptops) from ~25–35 to a stable 60.
  • Cut peak frame allocation from MB/s to KB/s (kill GC stutter).
  • Cut per-level startup hitch (currently builds 20–110 platforms up front).
  • Reduce GPU memory and draw-call count.

--------------------------------------------------------------------------------
B1. [P1] DRAW CALL EXPLOSION — every tile is its own material+geometry+mesh
--------------------------------------------------------------------------------
  main.js:495-597 createPlatform()
    Each tile builds a unique MeshStandardMaterial AND a unique
    BufferGeometry from makeArcGeometry. With segmentCount up to 13 and
    a level-1 target of 20 platforms (level 10 → 110 platforms), that's
    260–1430 separate draw calls just for tiles, each with its own material.

PROPOSAL (largest perf win in the codebase):
  1. Pre-build a SHARED arc geometry per (segmentCount, arcSize). Cache
     keyed by `${segmentCount}` since all tiles in the same platform share
     arcSize. Then each tile is just a Mesh referencing the shared geometry,
     rotated to its angle by `mesh.rotation.y = -i * arcSize` instead of
     baking the angle into vertices.
  2. Batch tiles by COLOR using THREE.InstancedMesh. One InstancedMesh per
     tile type (blue, gray, red, crackedBlue, finish, shop) per platform —
     or even per tower. This collapses hundreds of draw calls into a handful.
  3. Tile flash (updateTileFlashes) currently writes material.color directly.
     With InstancedMesh, switch to a per-instance color attribute
     (`InstancedMesh.setColorAt`). Update only the dirty instances.

INTERIM FIX (cheap, do this first):
  • Ensure tiles of the same type share a single material per platform.
    Today each tile gets `new MeshStandardMaterial`. Use a per-type material
    cache scoped to the platform (or global, with per-tile color override
    only when flashing). Roughly halves materials and draw calls without
    introducing instancing.

EXPECTED GAIN: 2–4× FPS on integrated GPUs.


--------------------------------------------------------------------------------
B2. [P1] Geometry & material cloning is rampant and unnecessary
--------------------------------------------------------------------------------
  Geometries are immutable buffers. Materials only need cloning when their
  per-instance state (color, emissive, opacity) actually diverges. Most of
  the clones in this codebase are defensive, not necessary.

  Sites that .clone() pointlessly:
    • main.js:452,461  spikeHoleGeometry.clone(), spikeHoleMaterial.clone(),
                       platformSpikeGeometry.clone(), platformSpikeMaterial.clone()
    • main.js:600     crateGeometry.clone(), crateMaterial.clone()
    • main.js:620     goldBlockGeometry.clone() — material can be shared if
                       flash is moved to a per-instance shader attribute,
                       but for now keep material clone (per-instance flash).
    • main.js:689     pillarSpikeGeometry.clone(), pillarSpikeMaterial.clone()
    • main.js:1195-1211  cannon parts: 4× geometry.clone() + 4× material.clone()
    • main.js:1591    sawBladeMaterial.clone()  (sawBladeGeometry already shared)
    • main.js:1625    pillarLaserRingMaterial.clone()
    • main.js:716     floaterMaterial.clone()
    • All the createXxxMesh() enemy builders that .clone() materials even
       when no per-instance state varies (e.g. bat wings).

FIX:
  • Stop cloning geometries. Period. Geometries are render-only buffers.
  • Only clone materials when the instance MUTATES color/emissive/opacity
    (e.g. flash effects). Mark the rest as shared.
  • Document at the top of main.js: "Materials below this line are SHARED;
    treat them as read-only at runtime."

EXPECTED GAIN: lower GPU memory pressure; faster level start; fewer GC hits.


--------------------------------------------------------------------------------
B3. [P1] Per-frame Vector3 / Material / Mesh allocations
--------------------------------------------------------------------------------
  Garbage in the hot path causes minor-GC stutter on low-end. Hot offenders:

  • spawnExplosion (main.js:2180) — creates `count` new MeshBasicMaterial
    per call. Called on every enemy death, every tile break (count=18),
    every shockwave (...). Easily 100s of new materials per second.
  • spawnBulletImpact (2195) — 7 new materials per bullet hit.
  • spawnGoldSparkle (2927) — 1 new material every 0.06–0.12s per gold
    block; 3 gold blocks/level → ~30/sec/level.
  • updateAcidPuddles inner loop (1023) — 2 new MeshBasicMaterial per puddle
    every 0.05s.
  • spawnFloatingText (3499) — new 256×128 canvas, 2D context, CanvasTexture,
    SpriteMaterial, Sprite per call. Fired on Reload, Combo Loss, NO HIT,
    SHIELD, COMBO SHIELD, VAMP +1.
  • updateComboSprite (1760) — same canvas-texture allocation EVERY combo
    increment.
  • bullet spawnBullet (2028) — `velocity.clone()` per shot.
  • getBounceCubeWorldPosition (2405) — `cube.mesh.position.clone()` every
    call (called twice per cube per frame).
  • isGoldBlockPositionClear (629) — new Vector3 per gold-block placement
    check, but worse: nested for-of loops also new Vector3 per gold block.
  • damageGoldBlock (2910), checkBallGoldBlockStomp (2835),
    checkBulletGoldBlockHit (2951) — `new THREE.Vector3()` per call.
  • spawnAcidPuddle (935) — fresh CircleGeometry per puddle (this one is
    legitimate because of varying size, but pool the two sizes).
  • detachCratesAndGoldFromTile (2477) — `new Vector3` inside both inner loops.
  • getArControllerDistance (4576) — 2 fresh Vector3 every poll.
  • handlePlatformCollision (4177), checkBulletPlatformHit (4105),
    handlePlatformUndersideCollision (4150) — `crossedPlatforms = []` per
    call, then sort. With the bandIndex it's usually 1 element so the sort
    is harmless, but the array allocation isn't.

FIX (general pattern):
  • Pre-allocate particle materials. Particles already share particleGeometry;
    extend the same idea: a shared MeshBasicMaterial per `color` value, with
    instance-level opacity using vertex colors via InstancedMesh, or simply
    accept that all particles of the same explosion share one material.
  • Pool MeshBasicMaterials keyed by color hex.
  • Replace floating texts and combo sprite with a SINGLE persistent canvas
    + SINGLE CanvasTexture; redraw into it and set texture.needsUpdate = true.
    Already 1 sprite per text — but allocate once, redraw on demand.
    Better: pre-render a small atlas of common strings ("Reload",
    "Combo Loss", "NO HIT", "SHIELD", "VAMP +1", digits 0–999) at boot.
  • Add module-level scratch Vec3s for the remaining sites; you already
    have 25+ scratch vectors at lines 3974-4003 — extend to cover the
    sites listed above.
  • Reuse `crossedPlatforms` array as a module-level scratch; clear with
    `arr.length = 0`.

EXPECTED GAIN: eliminates audible/visible "hiccups" on Android.

B4. [P1] Whole-level construction up-front
--------------------------------------------------------------------------------
  main.js:3656-3660 startLevel()
    for (let i = 0; i <= target; i += 1) createPlatform(...)
  Level 1 = 20 platforms; Level 10 = 110 platforms; each platform spawns
  enemies, cannons, and other props in createPlatform → maybeSpawnXxx.

  This means a multi-second hitch when a level starts, AND the player
  pays full memory + draw-call cost for content that's tens of meters
  below the camera.

FIX (streaming):
  • Generate only the platforms within ~12 spacings below the ball
    initially (≈ 2× the recyclePlatforms keep-zone above the player).
  • Track nextPlatformId and the deepest spawned Y; when the ball passes
    a threshold, spawn the next 1–2 platforms.
  • Pair this with a level-end "finish platform" placeholder: when the
    streamer is within N platforms of `target`, force the next spawn to
    be the final platform.
  • Same model for spawnPillarSpikesForLevel, spawnFloatersForLevel,
    spawnSawBladesForLevel, spawnPillarLaserRingsForLevel,
    spawnGoldBlocksForLevel — these currently iterate the full level.

EXPECTED GAIN: removes startup hitch; lower memory; better thermals on mobile.


--------------------------------------------------------------------------------
B5. [P1] Renderer settings are not adaptive
--------------------------------------------------------------------------------
  main.js:92-96
    new THREE.WebGLRenderer({ antialias: true, alpha: true });
    setPixelRatio(Math.min(window.devicePixelRatio, 2));
    shadowMap.enabled = true;

  • antialias:true forces 4×MSAA-equivalent fragment cost everywhere.
  • alpha:true is unused — scene.background is set, so drop it (frees a
    framebuffer attachment).
  • DPR cap of 2 on a Pixel 4a (DPR 2.75) means 22% more pixels than DPR 1.5.
  • shadowMap default is PCFShadowMap with default size (512×512). Many
    objects have castShadow = true, including small spikes, gold blocks,
    cannons, sawblades, enemies. Shadows are the most expensive part of
    a Three scene by a wide margin on integrated GPUs.

FIX (adaptive quality preset):
  Detect a "low" preset based on:
    - navigator.hardwareConcurrency <= 4
    - navigator.deviceMemory <= 4
    - DPR > 2
    - or a manual Quality option in the pause panel.
  Apply preset:
    LOW:
      antialias: false, alpha: false, dpr: 1.0,
      shadowMap.enabled: false (or 256×256 BasicShadowMap on key objects only),
      ball segments 16×12, pillar segments 24,
      hemisphere light only, drop directional sun's shadow,
      reduce particle count factor by 0.5,
      disable jellyfish/puffer transparency emissive,
      use MeshLambertMaterial for tiles (no metalness/roughness PBR).
    MEDIUM (default):
      current settings minus alpha:true, dpr cap 1.75.
    HIGH:
      antialias on, dpr cap 2, shadows.
  Persist the choice in localStorage.

B6. [P2] Material choices for opaque scenery
--------------------------------------------------------------------------------
  Almost every static prop (tiles, pillar, ledges, crates, gold) uses
  MeshStandardMaterial — physically based, expensive shader. The art style
  is flat-shaded cartoon; PBR is overkill.

FIX:
  • Replace tile/pillar/crate/gold materials with MeshLambertMaterial
    (cheap diffuse) or MeshPhongMaterial. On low preset use MeshLambert.
  • Keep MeshStandardMaterial only for the gold block (it benefits from
    metalness) and the player ball — and only on HIGH preset.

EXPECTED GAIN: 10–25% GPU time on integrated.

B9. [P2] Frustum / Y-distance culling for distant platforms
--------------------------------------------------------------------------------
  Platforms below the ball but within the recycle zone (~12 spacings) are
  always rendered; some are visible, some aren't (camera frustum). Three.js
  does frustum culling per Mesh, which is fine, but with hundreds of meshes
  that's hundreds of bounding-sphere tests per frame.

FIX:
  • Wrap each platform's tiles into a single Group with a precomputed
    bounding sphere; let Three frustum-cull the group as one unit.
  • Set `platform.group.visible = false` for platforms below
    ball.position.y - 30 (below the camera's far plane minus margin).
    Restoring visibility is a no-op when ball climbs back (it doesn't).

B10. [P2] scene.updateMatrixWorld(true) every frame
--------------------------------------------------------------------------------
  main.js:5182
    scene.updateMatrixWorld(true);
  Forces a recursive matrix update for the entire scene every frame.
  Three.js auto-updates matrices on render; calling this manually before
  render double-pays the cost.

FIX:
  • Remove the line. If a specific subtree needs early matrices for a
    same-frame collision query, do `world.updateMatrixWorld(true)` ONCE
    only on the subtree, only the frames where you actually query.
  • Better: set `mesh.matrixAutoUpdate = false` on static meshes
    (recycled platforms' tiles never move locally) and call
    `mesh.updateMatrix()` once when added.

B12. [P2] Wireframe shockwave = expensive

  high overdraw on a 24×16 sphere.

B13. [P3] Audio nodes are created per call
--------------------------------------------------------------------------------
  systems/audio.js — every play* function creates a fresh OscillatorNode
