PLAN DE REFACTOR DE POGOMACHINE

Objetivo
Separar src/main.js en modulos claros para que deje de ser un monolito. El nuevo main.js debe quedar como composition root: crear renderer/scene, crear estado, inicializar sistemas, registrar eventos y ejecutar el loop. La logica de enemigos, fisicas, plataformas, shooting, pickups, UI, tienda y leaderboard debe vivir en modulos dedicados.

Restriccion solicitada
En esta etapa solo se genera este documento. No se debe cambiar codigo fuente todavia.

Decision funcional importante
Eliminar el modo AR durante la refactorizacion. No moverlo a otro modulo. Debe desaparecer del producto para reducir complejidad, dependencias WebXR y ramas especiales del loop.

2. Problemas principales a resolver

Acoplamiento por estado global
Muchas funciones dependen directamente de variables globales como scene, world, ball, platforms, enemies, bullets, score, coins, hp, currentLevel, timeScale y DOM nodes. Si se extraen funciones sin una estrategia de contexto, se generaran imports circulares o listas enormes de parametros.

Orden de update implicito
El loop tiene un orden sensible: shooting, powerups, fisica de ball, scene matrix, colisiones, hazards, bullets, enemies, pickups/effects, recycle, camera. Ese orden debe conservarse durante la migracion.

Ownership de objetos Three.js
Cada sistema crea meshes/materiales y tambien los destruye. Si se separa mal, habra leaks o meshes huerfanos. El modulo que crea una entidad debe documentar como se destruye.

UI y gameplay mezclados
Varias acciones de gameplay actualizan DOM directamente. Ejemplos: score, coins, ammo, level progress, shop. El plan debe introducir una capa UI que reciba estado o eventos.

AR duplica mundo y render path
AR mantiene arScene, arRoot, arPlatforms, arBall, arBullets, WebXR session y branch especial en animate. Como se va a eliminar, conviene hacerlo antes de extraer el loop para reducir superficie.

5. Orden recomendado de migracion

Fase 0: Baseline y seguridad
- Crear una checklist manual de comportamiento antes de mover codigo: iniciar juego, rotar torre, disparar, shotgun, Steam Deck, pausa/opciones, tienda, leaderboard, completar nivel, morir/reset.
- Ejecutar npm run build antes y despues de cada fase.
- Hacer cambios pequenos y testeables, no intentar mover todo main.js de una vez.

Fase 1: Eliminar AR
- Aplicar la eliminacion descrita arriba.
- Verificar que no quedan referencias a arMode, arStatus, renderer.xr, arScene ni body.ar-active.
- Verificar que Steam Deck no dependia de AR.

Fase 2: Crear GameContext sin mover logica aun
- Crear src/game/context.js.
- Mover ahi el estado global y colecciones gradualmente, o crear un ctx que apunte a las variables existentes como paso intermedio.
- Objetivo: nuevas extracciones reciben ctx y no importan main.js.

Fase 3: Extraer render/assets
- Mover scene/camera/renderer/lights a src/render/scene.js.
- Mover geometrias/materiales a src/render/assets.js.
- main.js importa createSceneBundle y createGameAssets.
- Verificar que dispose sigue funcionando.

Fase 4: Extraer UI/DOM/events
- Mover selectors a src/ui/dom.js.
- Mover updateHeartsUI/updateCoinsUI/ammo/level/options a src/ui/hud.js.
- Mover listeners de botones a src/ui/events.js.
- Mantener actions en ctx para evitar que ui importe sistemas internos.

Fase 5: Extraer input normal y Steam Deck
- Dejar src/systems/input.js solo para input normal.
- Crear src/systems/steamDeckInput.js para fullscreen/pointer lock/gamepad/mouse held fire.
- Input escribe en ctx.input o llama actions publicas como ctx.actions.startShooting('steamDeckFire').

Fase 6: Extraer shooting y bullets
- Mover fireMachinegun, fireShotgun, fireCurrentWeapon, startShooting, stopShooting, updateShooting, updateBullets, clearBullets, reloadAmmo.
- Al principio puede seguir llamando collision callbacks desde ctx.collisions.
- Luego separar updateBullets en movimiento y collision processing si hace falta.

Fase 7: Extraer effects/pickups simples
- Mover particles, floatingTexts, shockwaves a render/effects.js.
- Mover coinPickups y bounceCubes a entities/pickups.js o submodulos.
- Verificar que score/coins se actualizan por actions centralizadas.

Fase 8: Extraer platforms
- Mover makeCrackLine/makeGrayCrackLines, createPlatform, tile color, spike trap creation, platform band index, getTileAtWorldPoint, recyclePlatforms, updateTileFlashes.
- Mantener callbacks para maybeSpawnEnemiesForSection y maybeSpawnCannon.
- Verificar generacion de nivel y shop tile.

Fase 9: Extraer enemies
- Primero mover mesh factories a enemies/meshes.js, es el cambio menos riesgoso.
- Despues mover spawn rules a enemies/spawn.js.
- Despues mover updateEnemies/positionEnemy/combat a enemies/update.js y enemies/combat.js.
- Mantener arrays en ctx.entities.enemies.

Fase 10: Extraer obstacles
- Mover cannons, saw blades, pillar spikes, pillar laser rings, spike traps.
- Si quedan dependencias cruzadas con collisions, pasar callbacks o services.

Fase 11: Extraer physics/collisions
- Mover ball physics y collision helpers.
- Mantener el orden del loop intacto.
- Este paso debe hacerse despues de platforms/enemies porque collisions necesita APIs claras de esos sistemas.

Fase 12: Extraer lifecycle/levels/shop/leaderboard
- Mover startLevel/resetGame/endGame/completeLevel a systems/gameLifecycle.js y systems/levels.js.
- Mover shop a systems/shop.js.
- Mover JSONBin leaderboard a systems/leaderboard.js.

Fase 13: Reducir main.js
- main.js deberia quedar solo con imports, create app, setAnimationLoop, resize cleanup si aplica.
- Eliminar funciones residuales y variables globales que hayan quedado sin usar.

7. Reglas para evitar un nuevo monolito

- Ningun modulo de entidad debe consultar DOM directamente.
- Ningun modulo UI debe mutar arrays de entidades directamente.
- Ningun modulo debe importar main.js.
- Evitar singletons globales nuevos. Preferir ctx pasado por parametro.
- El modulo que crea meshes debe exponer clear/dispose o documentar ownership.
- Mantener temporary vectors dentro del modulo que los usa para evitar allocations sin compartir demasiada memoria global.
- Evitar imports circulares: si A necesita llamar B, usar callbacks/actions en ctx.
- Refactorizar por movimiento mecanico primero; mejorar diseno despues de tener build verde.


8. Verificacion por fase

Comandos
- npm run build despues de cada fase.
- Si se agregan tests mas adelante, crear tests unitarios para utils puros: angleInArc, platform band index, getAngularDistance, spawn rules deterministas con RNG inyectado.

Checklist manual minima
- El juego carga y muestra HUD.
- Drag normal rota la torre.
- Click/release normal dispara segun modo actual.
- Steam Deck entra en fullscreen/pointer lock.
- Steam Deck rota con movimiento relativo.
- Steam Deck dispara manteniendo right trigger o click izquierdo.
- Options abre/cierra y modifica valores.
- Bullets impactan enemigos, plataformas grises, crates, gold blocks y cannons.
- Ball rebota en tiles azules, recibe damage en rojos y hazards.
- Shop abre en tile shop y compras funcionan.
- Level complete muestra rewards y next level.
- Game over y leaderboard siguen funcionando.


9. Riesgos principales

- Cambiar el orden del loop puede alterar gameplay aunque el build pase.
- Mover collisions antes de platforms/enemies puede multiplicar dependencias.
- Separar UI tarde puede hacer que sistemas extraidos sigan tocando DOM.
- Las funciones de dispose son faciles de romper y pueden dejar memory leaks.
- JSONBin access key esta en cliente; no es parte directa del refactor, pero queda como riesgo de seguridad existente.
- Existing src/core/state.js puede confundirse con el nuevo estado real. Conviene integrarlo o eliminarlo cuando se migre, no dejar dos fuentes de verdad.


10. Resultado final esperado

- main.js pequeno, legible y sin logica de gameplay interna.
- AR eliminado completamente.
- Sistemas separados por responsabilidad.
- Dependencias claras mediante ctx/actions.
- Mismo comportamiento gameplay que antes del refactor, salvo la eliminacion intencional del modo AR.
- Base mas preparada para agregar enemigos, powerups, modos de input o tests sin seguir inflando main.js.
