--------------------------------------------------------------------------------
0. GROUND RULES
--------------------------------------------------------------------------------
0.1  PlayCanvas is the source of truth for the SCENE. The Editor owns the
     hierarchy, materials, lights, render layers, and template (prefab)
     definitions. Scripts only INSTANTIATE templates and READ/WRITE entity
     transforms / properties.

0.2  VS Code is the source of truth for the SCRIPTS. We do not edit script
     bodies in the Editor's web editor; the VS Code extension syncs files in
     the project's `scripts/` folder.

0.3  No physics engine. There is no RigidBodyComponent, no CollisionComponent,
     no Ammo runtime. All collisions are math-based queries (sphere-sphere,
     sphere-AABB, swept-Y plane crossing) authored as plain JS in scripts.
     Reuse the algorithms from src/main.js verbatim; only swap the math
     primitives (THREE.Vector3 -> pc.Vec3).

0.4  Asset-only content. No CanvasTexture, no procedural geometry, no
     ExtrudeGeometry. Where the source generated something at runtime, we
     either:
       (a) replace it with an authored GLB / WebP, or
       (b) replace it with a PlayCanvas Element (text, image) for UI,
     never with runtime-built buffers.

0.5  Memory safety: PlayCanvas owns asset GPU lifetime. Scripts must NEVER
     call material.destroy() or model.destroy() on shared assets. Use
     `entity.destroy()` only on entities the script itself instantiated
     from a Template.

0.6  No pixel-ratio surprises: configure quality through a single
     `quality-controller.js` script attached to Root, exposing LOW/MED/HIGH
     attributes. Renderer-level toggles go through pc.app.graphicsDevice
     and pc.app.scene settings, not pc.app.renderer.

6.1 core/game-state.js (singleton)

6.2 core/game-loop.js
  Attached to: Root
  Attributes: none.
  Method: update(dt)
    Runs the same orchestration order as main.js animate() (lines 5137-5215):
      damageSlowdownTimer decay, scaledTime accumulation,
      world rotation lerp,
      shooting -> powerups -> ball physics -> collision queries
      -> spike traps -> pillar spikes -> saw blades -> laser rings
      -> bullets -> enemies -> floaters -> acid puddles -> shockwaves
      -> cannons -> gold blocks -> falling crates/gold -> coin pickups
      -> particles -> bounce cubes -> recycle platforms -> floating texts
      -> tile flashes -> camera.
    Calls into the relevant systems by direct app.X references rather than
    relying on PC's per-script update order.

  Why centralized: the original code depends on a strict order. Per-script
  update() in PC runs in registration order, which is brittle. Have one
  authoritative orchestrator.

7. UPDATE ORDER (per frame, owned by game-loop.js)
--------------------------------------------------------------------------------
const dt = clamp(realDt, 0, 0.033) * timeScale;
this.app.gameState.scaledTime += dt;

if (paused) {
  // still update camera-shake and UI? Keep pause cosmetic only.
  return;
}

worldRotator.tick(dt);

if (!gameOver && !levelComplete) {
  shooting.tick(dt);
  combat.tickPowerups(dt);
  ballPhysics.tick(dt);          // moves ball, runs platform/crate/gold/
                                  // cannon/saw collisions internally
  spikeTraps.tickAll(dt);
  pillarSpikes.tickAll(dt);
  sawBlades.tickAll(dt);
  pillarLaserRings.tickAll(dt);
  bullets.tickAll(dt);
  enemies.tickAll(dt);
  floaters.tickAll(dt);
  acidPuddles.tickAll(dt);
  shockwaves.tickAll(dt);
  cannons.tickAll(dt);
  goldBlocks.tickAll(dt);
  fallingDrops.tickAll(dt);
  coinPickups.tickAll(dt);
  particles.tickAll(dt);
  bounceCubes.tickAll(dt);
  platformSpawner.recycle(ball.position.y);
  floatingTexts.tickAll(dt);
  combo.tick();
  tileFlashes.tick(dt);          // iterates a Set, not all tiles
}

cameraShake.tick(realDt);        // unscaled
ballVisual.tick(realDt);

PlayCanvas's automatic per-script update is left unused (or prefixed with
guards) so this single pipeline owns ordering. To prevent double-tick:
each entity script's `update()` early-returns unless `app.gameLoop.tickedManually`
is false. Keep one channel, not two.

9. MATH-PHYSICS PORT (what to translate, line-by-line targets)
--------------------------------------------------------------------------------
Translation cheat-sheet (Three -> PlayCanvas):
  THREE.Vector3            -> pc.Vec3
  THREE.MathUtils.clamp    -> pc.math.clamp
  THREE.MathUtils.lerp     -> pc.math.lerp
  THREE.MathUtils.degToRad -> n * pc.math.DEG_TO_RAD
  vec.distanceToSquared(o) -> vec.distance(o) ** 2  (PC has only distance;
                              keep your own distSq via dx*dx+dy*dy+dz*dz)
  mesh.position.copy(v)    -> entity.setLocalPosition(v.x,v.y,v.z) or
                              entity.setPosition(v) for world
  mesh.getWorldPosition(t) -> entity.getPosition() (returns shared vec --
                              copy if you need to mutate)
  group.localToWorld(v)    -> group.getWorldTransform().transformPoint(v, v)
  group.worldToLocal(v)    -> mtxInv.transformPoint(v, v)  -- precompute
                              inverse from getWorldTransform()
  scene.add(mesh)          -> parent.addChild(entity)
  mesh.removeFromParent()  -> entity.parent.removeChild(entity) or
                              entity.destroy()

Algorithms to port unchanged:
  Ball Y integration + terminal velocity
  Plane-cross detection (previousY, currentY vs platformTop)
  Sphere-sphere distSq (enemy contacts, gold blocks, sawblades, cannons)
  Arc containment (angleInArc)
  Tile color flash lerp
  Spike raise piecewise function (getSpikeRaiseAmount)
  Cannon line-of-sight (narrow Y band + tile lookup)
  Bounce cube tangential motion in radial coords
  Shotgun spread tangent calculation
  Saw blade radial Y advance

DO NOT add pc.RigidBodyComponent. DO NOT add pc.CollisionComponent.
DO NOT call app.systems.physics.<anything>. The physics namespace is
unused.

Scratch vectors: declare module-statics in math-utils.js (pc.Vec3
instances) and reuse, mirroring main.js:3974-4003.

10. INPUT MAPPING
--------------------------------------------------------------------------------
PlayCanvas input system:
  Mouse  : this.app.mouse.on(pc.EVENT_MOUSEDOWN, ...)
  Touch  : this.app.touch.on(pc.EVENT_TOUCHSTART, ...)

11. UI PORT (Element vs HTML)
--------------------------------------------------------------------------------
Decision: use PlayCanvas Element components for the in-game HUD and panels.
Reasons:
  - Text/image rendering integrates with the render order.
  - Touch hits routed through PC's input system.
  - Avoids two parallel event pipelines (canvas + DOM).
Exception: leaderboard panel may use a small DOM overlay because it needs
a text input field; PlayCanvas Element doesn't ship a focused text-input
widget. Use a single div over the canvas for the name input only.

Sprite atlases:
  atlas_floating_text.webp drives floating-text.js.
  atlas_combo_digits.webp drives combo.js (assemble multi-digit combos by
    placing digit Elements left-to-right under a parent Element).

12. AUDIO PORT
--------------------------------------------------------------------------------
Route every play* call through audio-manager.js.

Implementation:
  AudioEntity has a SoundComponent with ~16 slots (one per role). Each slot
  has its asset, loop=false except invuln_loop=true, overlap=true so
  rapid-fire shoots don't truncate.

  audioManager.play(role) does:
    const slot = this.entity.sound.slot(role);
    if (slot) slot.play();
  audioManager.startInvulnLoop():
    const s = this.entity.sound.slot('invuln_loop');
    if (s && !s.isPlaying) s.play();
  audioManager.stopInvulnLoop():
    fade gain via slot.volume animation, then s.stop().

The procedural oscillators in src/systems/audio.js are deleted entirely.

17. RISKS & OPEN QUESTIONS
--------------------------------------------------------------------------------
R1  PC's per-script update order: solved by central game-loop.js but
    requires discipline -- per-script update() must not duplicate work.

R2  Templates instantiating with all child Element / Render assets cold
    on first use can still hitch on mobile. Mitigation: at boot, instantiate
    one of each template, parent under a hidden warmup entity, then destroy.
    Triggers GPU upload + JIT before gameplay.

R3  Sprite atlases require an authoring step. Provide a small Node script
    that renders the strings to PNG, runs through cwebp; track in docs/.

R4  PlayCanvas WebXR coverage on iOS Safari is limited; AR likely
    Quest-only at parity. Acceptable.

R5  No Ammo physics means the team must accept that "physics-feel" tweaks
    are still code edits, not Editor knobs. Document the math constants on
    balance.js for designer access.

R6  Leaderboard requires a Firebase project + billing review (free tier
    is fine for this scale). Spend 1h on rules/auth before going live.

R7  GLB authoring overhead: ~30 props to model. Identify reusable atomic
    shapes (single tile arc, single particle sphere) and fan out via
    materials + transform.
